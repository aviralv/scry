# 2026-05-15 — Project Kickoff

## Context

Emerged from a research session exploring Glean, Notion CLI, and markdown-based team knowledge systems. The research surfaced that the "personal cross-tool search" problem is unserved at small-team scale, and that MCP makes the connector layer trivial — the missing piece is orchestration + synthesis.

## Decisions

- **Name**: scry ("to see what's hidden")
- **Architecture**: Search orchestration, not a search engine. Use native search APIs via MCP, don't re-index content.
- **Context registry**: YAML file of people/projects/source mappings as the routing intelligence
- **No RAG**: Don't fetch full content at query time. Synthesize over search snippets returned by each tool's API.
- **Stack**: TBD (Python likely given MCP server ecosystem, but deferred)
- **Repo**: Private. Session notes live inside.

## Architecture (Validated)

Gemini 2.5 Pro critiqued a "metadata index + live fetch full content" design as too slow. But that's not what we're building — we're calling native search APIs that already return snippets. The orchestration pattern is:

1. User query → consult registry → identify relevant sources/channels/people
2. Fan out parallel MCP search calls (slack_search, confluence_search, etc.)
3. Each returns ranked results with previews (~2-3s total in parallel)
4. LLM synthesizes across snippets with source attribution
5. Deep fetch only on explicit user request

## First Milestone

Query across Slack + Confluence + email → synthesized answer with citations → faster than manually searching each tool.
